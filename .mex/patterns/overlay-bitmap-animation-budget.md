# Overlay Bitmap Animation Budget

Full bitmap animation data must be budgeted separately from overlay code.

- Executable overlays run from `ring_buffer` and each packed `SPCTLK*.OVL` block is limited to 2048 bytes.
- A frame-data stream larger than one block cannot be `INCBIN`ed into an overlay that must keep executing after load.
- The original 16-frame GloboTerraqueoZX delta+logo package was already far over budget for a direct `SPCTLK2` `INCBIN`: `earth_frame0.compact.bin=584B`, `earth_frame_deltas.bin=6189B`, `earth_attr0.compact4.bin=44B`, `earth_attr_deltas.compact4.bin=555B`, `earth_logo.bin=528B`. The standalone demo was about `9020B`.
- The accepted `!about` shape imports assets under `release/about_earth/`, stores the binary payload in `SPECTALK.DAT`, and keeps `SPCTLK2` to executable code plus mutable compact buffers.
- The reduced logo is 176x24 (`22` bytes wide, 3 attr rows). For SpecTalk's about layout it is drawn at pixel Y=104 (`EARTH_LOGO_SCREEN_ADDR=$48A5`, attrs row 13..15), leaving rows 16..17 for Ikkle text and row 20 for the normal-font footer.
- If the animation is kept, store frame data in `SPECTALK.DAT` before help and read chunks into `overlay_slot`; reducing resolution/frame count or converting to a procedural hybrid remain fallback options if overlay code grows again.
- `SPECTALK.DAT` is a viable home for large cold animation data because startup only reads the fixed font/theme/BPE prefix. Do not append raw binary after help without a length/sentinel: help currently treats EOF as the end of text. Use `font+themes+BPE dict + globe block + help`, with generated `EARTH_*_OFFSET/SIZE` and `BPE_HELP_OFFSET` constants.
- For streaming cyclic SKIP/COPY deltas, do not keep separate bitmap and attr file streams. Generate one interleaved tick stream with length prefixes: `uint16 frame_delta_len + frame_delta + uint8 attr_delta_len + attr_delta`, repeated for each frame. The divMMC release path pads packets only to the generated maximum packet size (`481B` in the current asset pass); 512B sector packets belong only to the parked Next `DISK_FILEMAP` experiment.
- Any hand-written ASM that consumes the generated Earth stream must take `EARTH_PACKET_SIZE`, `EARTH_DELTA_OFFSET`, and frame count from the same generator as `overlay_api.h`. Do not leave private `DEFC` copies stale: a `512B` ASM read against `481B` DAT packets advances 31 bytes into the next packet, so the next tick can parse a bogus delta length and apply deltas outside the valid `overlay_slot` packet. `tools/bpe_build.py` now patches `overlay_entry2.asm`; keep it in `BPE_INPUTS` and in the BPE save/restore set.
- Delta packages still need mutable frame/attribute buffers in RAM before applying SKIP/COPY deltas; do not count only the static delta stream.
- For `!about`, keep the existing `OVERLAY_ABOUT` model on divMMC: normal IRC parsing is paused after the overlay is loaded because `ring_buffer` holds animation code, `overlay_keepalive()` answers server `PING` and preserves automatic NAMES state, and exit must not add a fixed input-blocking drain.
- Measure both `SPCTLK2.OVL` size and perceived animation speed. Current divMMC baseline after parking `DISK_FILEMAP`: `SPCTLK2.OVL=1860B`, `SPECTALK.DAT=15155B`, `SPECTALK.OVL=10240B`; resident remains tight. About rows, separator, logo, and Earth attributes must adapt to the active theme; only theme 1 keeps the deliberate black/multicolor identity. Apply bitmap and attr deltas before drawing each tick so color movement is not one frame stale. Public ASM entries reached via `overlay_call` must preserve `IX`; tail-calling a renderer that leaves `IX` on span metadata can corrupt the C caller after the animation tick.
- The 24-frame west-to-east pass with `TEMPORAL_HYSTERESIS=0.025` produces `earth_frame0.compact.bin=587B`, polished `earth_frame_deltas.bin=9159B`, `earth_attr0.compact4.bin=44B`, `earth_attr_deltas.compact4.bin=776B`, `earth_logo.bin=528B`, and an `Earth 12703B` block in `SPECTALK.DAT` with 481B packet padding. The `tools/about_earth_polish.py` pass fills only single black pinholes fully surrounded by lit pixels and leaves colors/attributes unchanged. The main loop currently advances the globe every other frame to cut SD I/O and reduce temporal-dither shimmer. The logo renderer preloads 23 rows into `overlay_slot` plus one final row read instead of reading all 24 rows separately.
- For cyclic streams, keep the DAT handle open and rewind with esxDOS `F_SEEK` when the frame counter wraps. Close/open plus skip-by-reading works but adds recurrent SD I/O and can reintroduce long-term clock drift.
- If a tiny static dither sparkle from the source render looks wrong on hardware, a post-draw screen-byte mask is cheaper and less invasive than regenerating/repacking the whole `SPECTALK.DAT` payload, as long as the mask is limited to known fixed pixels and re-applied after every frame draw.
- The post-draw lower-edge sparkle mask is still optional. DivMMC now has enough `SPCTLK2` headroom to reintroduce it, but only after the current `/names`/timeout hardware pass proves the protocol path stable.
